## TL;DR

AIエージェントにcronでタスクを繰り返させると、**完了条件がないために同じチェックを無限ループする**問題が起きる。解決策は3つ：①タスクに明示的な完了条件を書く、②実行回数の上限を設ける、③冪等性（idempotency）を保証する。実際にRevenueCat APIを何10回も同じ結果で呼び続けたバグの修正事例を紹介する。

## 前提条件

- AIエージェント基盤（OpenClaw, LangChain, CrewAI等）でcron/定期タスクを運用中
- エージェントが自律的にAPI呼び出し・状態チェックを行う構成

## 問題：Heartbeatが同じAPIを無限に叩き続けた

自律AIエージェントに「Revenue Watchタスク」を設定していた。内容は単純で、RevenueCat APIでMRR・購読者数を取得して報告するもの。

```
# HEARTBEAT.md（問題のあった設定）
## Revenue Watch
- RevenueCat APIでMRR確認
- 異常があればSlackに通知
```

結果：エージェントはRevenueCat APIを**30回以上**呼び出した。毎回MRR $28、購読者5名という同じ結果を取得し、「次のステップ」として同じAPIチェックに戻った。**完了条件がないため、エージェントは「まだ終わっていない」と判断し続けた。**

トークン消費量：通常のheartbeatの10倍以上。

## 根本原因

| 原因 | 詳細 |
|------|------|
| 完了条件の欠如 | 「確認する」だけで「確認したら終了」がない |
| 状態の非保持 | 前回の実行結果を記憶しないため、毎回「初めて」のように動く |
| 暗黙の継続 | エージェントは「タスクリストが空になるまで続ける」のがデフォルト動作 |

## Step 1: 完了条件を明示的に書く

```markdown
# 修正後のHEARTBEAT.md
## Revenue Watch
- RevenueCat APIでMRR確認
- **完了条件: APIレスポンスを1回取得し、結果をログに書いたら終了**
- 異常時のみSlackに通知（正常時は通知不要）
```

ポイントは「1回取得したら終了」という**明示的な停止条件**。

## Step 2: 実行回数の上限を設ける

タスク定義に`max_iterations`や`max_api_calls`を含める。

```markdown
## Revenue Watch
- max_iterations: 1
- RevenueCat APIでMRR確認
- 1回のAPI呼び出しで結果を取得→報告→終了
```

フレームワークレベルでも制御できる：

```python
# LangChainの場合
agent = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=5,  # 5回で強制停止
    early_stopping_method="force"
)
```

## Step 3: 冪等性を保証する

同じタスクを2回実行しても、副作用が1回分しか起きないようにする。

```markdown
## Revenue Watch
- 前回の実行結果: {last_result}
- 前回と同じ結果なら「変化なし」と報告して即終了
- 変化があった場合のみ詳細を報告
```

前回の結果をファイルやDBに保存し、差分がなければ早期終了する。

## Step 4: ガードレールを追加する

```bash
# cronジョブのタイムアウト設定（60秒で強制終了）
timeout 60 openclaw heartbeat run
```

システムレベルのタイムアウトは最後の砦。エージェントの自律性を信頼しつつ、暴走時の安全弁を用意する。

## 修正結果

| 指標 | 修正前 | 修正後 |
|------|--------|--------|
| API呼び出し回数/実行 | 30回以上 | 1回 |
| トークン消費 | 通常の10倍 | 通常の1倍 |
| 実行時間 | 数十分（ループ） | 30秒以内 |
| 正確性 | 同じ結果の繰り返し | 変化のある時だけ報告 |

## まとめ

| 教訓 | 詳細 |
|------|------|
| 完了条件は省略できない | 人間は暗黙に「1回やったら終わり」と理解するが、エージェントはそうではない |
| max_iterationsは必須 | 完了条件のバグに対する安全弁 |
| 冪等性で無駄を排除 | 前回と同じ結果なら即終了 |
| タイムアウトは最後の砦 | システムレベルで暴走を止める |

AIエージェントのcronジョブを設計するとき、「何をするか」だけでなく「いつ止まるか」を必ず定義すること。これを怠ると、エージェントは永遠に同じことを繰り返す。
